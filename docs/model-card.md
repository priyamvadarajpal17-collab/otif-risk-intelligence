# Model Card — OTIF Risk Intelligence

This model card documents the realistic OTIF risk-intelligence
control tower prototype: a noisy, partially-observable synthetic supply-chain digital
twin, an order-level XGBoost risk model, a 10-node mechanism Bayesian network with
structural-intervention scenario analysis, evidence-based fusion, resource-aware
interventions, and a local daily operating loop with drift detection and versioned
retraining.

All numbers below are **measured, not asserted**: generated by
`uv run otif-benchmark --seeds 1 2 3 4 5 --orders 2500 --output-dir artifacts`,
`uv run otif-policy-benchmark --seeds 1 2 3 4 5 --orders 2500` (Decision Value Lab), and
`uv run otif-risk --orders 2500 --seed 42 --output-dir artifacts` on this codebase.
Artifacts themselves are gitignored; this document summarizes what a fresh run produced.

## Intended use and scope

- A **demonstration prototype** for a control-tower-style OTIF (on-time-in-full) risk
  desk: one risk score and one intervention decision per open order, with affected-SKU
  evidence, a mechanism-split causal pathway explanation, structural-intervention
  scenarios, and resource-aware conflict handling.
- Trained and evaluated entirely on a **synthetic digital twin** (`src/otif_risk/data.py`);
  it is not fitted on, or validated against, real operational data.
- Not a production forecasting service: no live ERP/WMS/TMS integration, no
  cloud deployment, no message broker, no feature store. The optional AI Copilot layer
  (below) is the one place this prototype can call a real external API (OpenAI), and only
  to explain/draft text -- it never scores, decides, or writes back to the pipeline.


## Data: the noisy digital twin

Unlike a generator that pre-selects which orders miss and forces matching events, this
twin simulates each order's lifecycle stage by stage (capture, vendor readiness, line
allocation, warehouse, transport, customer delivery) from:

- **stable heterogeneity**: persistent vendor reliability/variance, SKU criticality and
  scarcity traits, DC capacity variability, lane transit variability, customer
  appointment/reschedule traits;
- **seasonality**: a smooth demand/congestion curve across the simulated calendar;
- **correlated shocks**: randomly-timed vendor/DC/lane disruption windows with their own
  severity, plus a scripted late-horizon "concept-drift epoch" of elevated vendor
  disruption density;
- **partial observability**: ~3% of vendor/warehouse/transit events never post at all;
  exception codes are logged with ~85-90% probability even when the underlying delay
  crossed its threshold; a capture-time inventory shortfall has a documented chance of
  being backfilled by an expedited replenishment before shipment, so early evidence is a
  genuine *risk factor*, not a certainty;
- **measurement noise**: gamma/normal noise at every stage, not fixed offsets.

OTIF outcomes fall out of the **accumulated** per-stage delay and the **final** shipped
quantity — never chosen up front. Simulator truth (latent shocks, per-line
shortfall/affected flags, accumulated delay, shock identifiers) is persisted separately
(`simulator_truth.csv`, `line_truth.csv`, `shocks.csv`) and is used only for evaluation,
never as a model feature.

Five deterministic named scenarios are reserved (last 5 orders of every run, regardless
of seed): multi-cause propagation, two orders contesting shared recovery capacity,
a line-level (not all-lines) stockout, and a genuinely unexplained ("UNKNOWN") miss. The
remaining orders stay fully seed-random.

**Measured OTIF miss rate** across 5 seeds (2,500 orders each): median **17.5%**
(range 15.2%–20.2%), inside the 15–25% target band on every seed tested.

## Features and point-in-time contract

Every feature is a function of data knowable at (or before) an explicit
`as_of_timestamp` — the same contract used for historical snapshots
(`as_of_timestamp=None`, defaults to each order's own `prediction_timestamp`) and for the
daily operations replay (`as_of_timestamp=<simulated day>`). Rolling vendor/DC/lane/
customer/SKU history is computed in three matured windows (30-day, 90-day, all-time) via
vectorized as-of joins, plus freshness (`hours_since_last_observed_event`), missingness
(`missing_event_stage_count`), remaining slack (`remaining_slack_hours`), and a DC
utilization trend. Vendor rolling reliability is fault-attributed
(`vendor_rolling_fault_rate_*`), not raw OTIF-miss rate, so a vendor is not penalized for
DC/transport/customer-caused misses. `tests/test_features.py` proves that mutating every
future event/outcome cannot change an earlier snapshot's row, and that the chronological
60/20/20 split groups identical timestamps together rather than cutting mid-group.

## Line/SKU evidence

`line_evidence.py` scores every order line from capture-time-safe fields only
(initial ATP allocation, inventory snapshot, SKU criticality, the order's own
point-in-time vendor-exception signal) and flags lines as "likely affected." Measured
on the seed-42 canonical run's held-out test lines:

| | precision | recall |
|---|---|---|
| Targeted line evidence | **0.58** | **0.66** |
| Naive "flag every held-out line" baseline | 0.09 | 1.00 |

Targeted evidence is ~6.5x more precise than the naive baseline at comparable recall.

## Models

**XGBoost order-risk model** (`model.py`): gradient-boosted binary classifier over
~100 point-in-time features, Platt-scaled on validation.

**10-node mechanism Bayesian network** (`bayesian.py`): matches the actual OTIF
definition -- "on time" AND "in full" -- instead of one flat seven-cause star:
`ORDER_CAPTURE→LATE_DELIVERY`, `VENDOR_FAILURE→INVENTORY_SHORTAGE`,
`{INVENTORY_SHORTAGE, DC_CAPACITY}→WAREHOUSE_OPS`, `WAREHOUSE_OPS→TRANSPORT`,
`INVENTORY_SHORTAGE→IN_FULL_FAILURE`,
`{ORDER_CAPTURE, WAREHOUSE_OPS, TRANSPORT, CUSTOMER_DELIVERY}→LATE_DELIVERY`,
`{IN_FULL_FAILURE, LATE_DELIVERY}→OTIF_MISS`. `IN_FULL_FAILURE`/`LATE_DELIVERY` are fit
directly from each training order's own `1 - in_full`/`1 - on_time` outcome. CPTs are
smoothed counts fit on training-only operational stage history, including disruptions
absorbed by orders that still achieved OTIF. A node is only given as *hard* evidence
once its stage has actually been observed as of the as-of timestamp; unobserved
intermediate stages (e.g., `WAREHOUSE_OPS`/`TRANSPORT` before those events post, and the
two mechanism nodes themselves) are marginalized out via exact inference, so an early
vendor-failure signal can raise OTIF risk before the warehouse/transport stages have even
happened. Inference is exact either way: `pgmpy` variable elimination when available, or
a brute-force joint enumeration over this small 10-node binary network (verified
numerically identical in `tests/test_bayesian.py`) when it is not.

Pathway output includes the active evidence, the mechanism route(s) (a shared upstream
cause like `VENDOR_FAILURE` or `INVENTORY_SHORTAGE` can have routes to *both*
`IN_FULL_FAILURE` and `LATE_DELIVERY`), the two mechanism posteriors, the posterior/prior
risk, evidence coverage, a `LOW`/`MEDIUM`/`HIGH` confidence band, and an explicit
qualification that this is "probabilistic association within a fixed chain structure, not
a proven causal mechanism."

**Structural intervention scenarios and evidence attribution** (`bayesian.py`,
`BayesianBundle.intervene`): for every scored order, an exact
`P(OTIF_MISS | evidence, do(node=value))` is computed for each active evidence node
(mitigated to 0) plus one combined-mitigation scenario, always via brute-force
enumeration (never through the pgmpy observational path). A do-operator intervention
replaces the node's fitted CPT with a fixed value, severing its parents' influence on it
-- this is verified to differ from simply conditioning on the same value at a collider
node (`WAREHOUSE_OPS`, with parents `INVENTORY_SHORTAGE` and `DC_CAPACITY`) in
`tests/test_bayesian.py::test_intervention_severs_parent_influence_and_can_differ_from_conditioning`.
Each scenario reports baseline/post-intervention Bayesian posteriors, absolute/relative
risk reduction, and an explicit "Fixed-structure scenario analysis -- not a proven
treatment effect" qualification; these never feed the XGBoost score, the fused score, or
the operational decision. Leave-one-evidence-out **evidence attribution** reports, for
each active cause node, how much the posterior would change if that node were withheld
(marginalized) instead of conditioned on -- labeled `evidence_attribution_leave_one_out`,
explicitly not SHAP and not a causal-effect estimate.

## Fusion

`fusion.py` compares XGBoost-only, Bayesian-only, the fixed 70/30 blend, and every other
convex weight in 10% increments — one line search, no stacking model. The weight is
selected on **validation only**: candidates must keep top-15%-capacity recall within
0.03 of the best candidate and Brier score within 0.002 of the best eligible candidate.
Among those practically equivalent candidates, the policy prefers more Bayesian
contribution. The final decision threshold is then tuned separately for the chosen weight.

On the 5-seed benchmark, validation selected 20–30% Bayesian contribution; the seed-42
canonical run selected 10%. Median Bayesian PR-AUC is 0.475 and fused median PR-AUC is
0.705. The full comparison table (`fusion_comparison.csv`,
`metrics.json.fusion_comparison`) is always persisted.

## Decision Value Lab (Stage 1: measuring intervention value)

All numbers in this section come from
`uv run otif-policy-benchmark --seeds 1 2 3 4 5 --orders 2500` writing
`artifacts/policy_benchmark.json` (alongside its own `policy_benchmark_manifest.json` run
manifest). See the README's "Decision Value Lab" section for the full design
(action-response probability formula, capacity engine, headline-metric normalization,
the rolling-origin out-of-sample scoring design, and the value-aware `CURRENT_POLICY`
formula).

Every order is scored via rolling-origin, chronological cross-fitting (5 equal folds;
the first is warm-up history, excluded from evaluation; every later fold is scored only
by a model trained -- and its fusion weight/threshold selected -- on history strictly
before it). Coverage across the 5 benchmarked seeds is ~79.8-80.0% of the calendar (the
excluded first-fold warm-up is the remainder); every fold's own threshold is recorded in
each run's `scoring_coverage`.

Eight policies are compared: `NO_ACTION`, `RANDOM_AT_CAPACITY`, `HIGHEST_RISK_AT_CAPACITY`,
`HIGHEST_FINANCIAL_AT_CAPACITY`, `STRONGEST_SIGNAL_HEURISTIC`,
`SINGLE_CAUSE_PRIORITY_BASELINE` (the transparent `decisions.recommend_orders`
risk x tier x value ranking), `CURRENT_POLICY` (the **value-aware** deployable
policy -- see README for the exact formula: candidate actions from active point-in-time
leading signals and persisted Bayesian structural intervention scenarios, ranked by
expected avoided penalty per normalized resource capacity consumed), and
`ORACLE_EVALUATION_ONLY` (evaluation-only ceiling).

### Capacity-stress sensitivity analysis

At this twin's *default* (100%) daily capacities, resource pools are rarely binding for
the genuinely out-of-sample-thresholded eligible population under the single-action
baselines: on 3 of the 5 benchmarked seeds, every single-action deployable policy
(including plain `RANDOM_AT_CAPACITY`) selects the exact same set of orders, because
there is essentially no contention to prioritize among. Where a ranking has nothing to
contend over, it cannot outperform any other ranking -- this is a genuine, measured
property of this twin, not a bug, but it also means "does a policy beat simpler baselines
at 100% capacity" measures the wrong thing for a resource-constrained business.

`resources.CAPACITY_SCENARIOS` re-runs every policy (including the oracle) at three
pre-specified capacity multipliers applied uniformly to every resource pool:
`SCARCE_25_PERCENT` (0.25x), `SCARCE_50_PERCENT` (0.5x, **the Stage 1 headline scenario
every acceptance gate below is measured against**), and `BASE_100_PERCENT` (1.0x,
retained as a diagnostic). Continuous (`dc`/`lane`) pools scale directly; discrete
(`vendor`: 1 slot/day, `customer`: 2 slots/day by default) pools that would round to a
fractional slot below one whole unit use a deterministic whole-slot schedule (an
error-diffusion accumulator walked once across the evaluated calendar) so a 1-slot pool
at 50% realizes exactly `0, 1, 0, 1, ...` and at 25% realizes a slot on exactly one day
in four -- the same schedule shared, unmodified, across every policy.

**Discriminativeness rises sharply as capacity shrinks** (median contested rate / median
capacity-binding rate). `CURRENT_POLICY`'s rates are *lower* than `SINGLE_CAUSE_PRIORITY_BASELINE`'s
at every scenario -- an expected side effect of spreading eligible orders across more
resource pools instead of concentrating every order on the one pool its `primary_cause`
alone implies:

| Capacity scenario | Median contested rate (`CURRENT` / `LEGACY`) | Median capacity-binding rate (`CURRENT` / `LEGACY`) |
|---|---:|---:|
| `BASE_100_PERCENT` (diagnostic) | 0.4% / 1.4% | 6.0% / 12.0% |
| `SCARCE_50_PERCENT` (**primary**) | 3.9% / 7.9% | 43.0% / 45.9% |
| `SCARCE_25_PERCENT` | 27.1% / 30.4% | 43.0% / 45.9% |

### Bayesian-ablation diagnostic: does the Bayesian term add value?

`bayesian_ablation_diagnostic` compares `CURRENT_POLICY` with vs. without its Bayesian
structural-reduction term, holding every other formula component, the eligible pool, and
the capacity scenario fixed -- reported honestly, not required to favor the Bayesian
term:

| Variant | Median avoided penalty per normalized resource unit (50% capacity) | Seeds where it wins |
|---|---:|---:|
| With Bayesian term (deployed `CURRENT_POLICY`) | 9.964 | 0 / 5 |
| Without Bayesian term (leading-signal-fallback only) | 11.345 | 5 / 5 |

In this twin, the ablated variant *without* the Bayesian term scores higher at every
seed (median delta -1.227). This does not change the gate result -- `CURRENT_POLICY` is
defined to use the Bayesian term, and it still clears every primary-gate condition above
-- but it is an honest finding, not hidden: the persisted Bayesian structural scenarios
are not currently pulling their weight in this formula/twin combination. Carried forward
as an open question for Stage 2's champion/challenger tooling, which should be able to
promote either value-aware variant on its own merits rather than assume the
Bayesian-informed one is automatically better.

| Metric | `bayesian` vs. best | `strongest_signal` vs. best | `random` vs. best |
|---|---:|---:|---:|
| Mean top-action agreement (range across seeds) | 0.126–0.139 | 0.167–0.200 | 0.147–0.192 |
| Mean value regret (range across seeds) | 1.69–2.80 | 1.58–2.44 | 2.78–3.60 |

The Bayesian top structural intervention beats a random feasible action's mean value
regret in every seed, and beats the strongest-signal heuristic's mean value regret in 3
of 5 seeds (seeds 2 and 5 are the exceptions: Bayesian regret 2.48 vs.
strongest-signal's 1.58 on seed 2, and 2.80 vs. 2.44 on seed 5). Its raw top-action
**agreement** rate with the twin's counterfactually-best action is consistently *lower*
than the strongest-signal heuristic's, in all 5 seeds. Read together: the Bayesian
ranking adds measurable value (lower regret) without recovering the single best action
as often as a much simpler heuristic -- it is not forced to win, and it does not win on
every axis. This diagnostic does not depend on resource capacity or the fused
threshold, so it is unaffected by the capacity-stress scenarios above.

### What is (and is not) measured here

- The action-response probability/magnitude formulas, the capacity engine
  (`resources.py`), `decisions.py`'s existing priority formula, and the three
  capacity-stress multipliers were fixed **before** any benchmark run; no seed, weight,
  or baseline was retuned after seeing results -- including after seeing that the
  primary 50%-capacity acceptance gate fails.
- Every evaluated order is scored out-of-sample via rolling-origin cross-fitting (see
  above); coverage and every fold's threshold are reported in `scoring_coverage`, not
  silently assumed to be 100%.
- Every policy, including `ORACLE_EVALUATION_ONLY`, is evaluated at all three
  pre-specified capacity scenarios with the identical realized capacity for the same
  scenario/day (see `resources.build_capacity_schedule`); no policy is ever compared
  against a capacity another policy did not also see.
- `ORACLE_EVALUATION_ONLY` is an evaluation-only, unattainable ceiling (it may pick any
  action, ignoring the cause the current system would have inferred); it is never a
  deployable recommendation and is used solely to compute regret.
- This still evaluates against the synthetic twin's own potential outcomes, not observed
  real-world treatment effects -- see "Honest limitations" below.

## Governance (Stage 2): manifests, ledger, promotion, monitoring

All numbers below are from the canonical `uv run otif-ops --orders 2500 --seed 42
--replay-days 90 --policy-value-reference-path artifacts/policy_benchmark.json` replay
(see the README's "Governance (Stage 2)" section for full design detail).

- **Run manifest**: `run_manifest.json` checksummed 121 artifacts for the 90-day replay
  (29 for the canonical single-seed pipeline run) and verified clean
  (`manifest.verify_manifest` → `verified: true`). `content_id` is deterministic and
  excludes timestamps/run-instance/output-path fields; `test_manifest.py` proves two runs
  of identical seed/config/code produce the same `content_id` and that tampering with a
  checksummed artifact after the manifest is written is detected.
- **Offline/batch parity**: the canonical pipeline run's `parity_check.json` scored the
  same 15 orders at the same as-of snapshot directly (offline) and through the
  adapter/service boundary (`adapters.py` reading CSV-round-tripped tables); both
  produced byte-identical feature vectors and scores (`passed: true`).
- **Decision ledger and observational cohorts**: 6,226 decisions logged over the replay,
  5,923 reconciled against matured outcomes (303 still open at replay end). Cohort miss
  rates: `ACCEPTED` 57.96% (n=716), `MONITORED` 10.08% (n=5,206), `REJECTED` withheld
  (n=1, below the 20-decision minimum-sample guard). Each row carries its order's real
  `order_value`/`penalty_rate` (from `decisions.attach_business_context`), so
  `realized_penalty` is a genuine dollar figure, not a placeholder zero: mean realized
  penalty is $113.54 for `ACCEPTED` vs. $13.33 for `MONITORED`.
  `intervention_outcomes.json` further breaks realized outcomes down by the specific
  intervention type taken (e.g. `INVENTORY_REALLOCATION`: n=431, 70.3% miss rate) against
  a no-intervention baseline. The report is explicitly labeled
  `observational_not_causal: true` -- this gap reflects which orders the policy
  prioritized (correctly, higher-risk ones), since this replay's lifecycle does not yet
  feed action back into the twin; the Decision Value Lab above remains the only
  causally-interpretable measured policy value in this codebase.
- **Champion/challenger promotion**: of 9 trained model versions (1 initial + 8 real
  retrains), **every real retrain challenger was held** -- each retrain's own held-out
  test PR-AUC (on that retrain's necessarily small, recent slice of matured history in
  this 90-day window) regressed beyond the 0.02-absolute tolerance versus the initial
  champion's 0.717 (e.g. v5 0.509, v7 0.471, v9 0.343). The active model never advanced
  past `v1` for the entire replay. A separate demo governance-lifecycle scenario, built
  only from Stage 1's own measured `policy_benchmark.json` numbers (never fabricated) and
  explicitly labeled to keep it distinct from the real retrain lifecycle above,
  demonstrates all three required transitions: the value-aware action policy without
  Bayesian ranking **promoted** over the single-cause baseline (11.345 vs. 9.174 median
  policy value), a Bayesian action-ranking challenger **held** (9.964 vs. 11.345, a
  −12.2% regression beyond the 5% tolerance), and the active pointer **rolled back** to
  `v1`, a real, manifest-verified version.
- **Monitoring/SLOs**: soft data-quality checks (completeness, uniqueness, referential
  health) passed with zero contract failures; measured **local-only** runtime averaged
  1.48s/day scoring and 3.35s/retrain (explicitly not a production-latency claim). One
  SLO failed honestly: rolling realized PR-AUC (0.386) missed its 0.55 target -- the
  direct, reported cost of the promotion gate correctly keeping the safer `v1` champion
  active rather than promoting any noisier later retrain (see "Honest limitations"
  below).

## AI Copilot: read-only explanation/drafting layer

A grounded LLM copilot (`copilot_context.py`/`copilot_fallback.py`/`copilot_validation.py`/
`llm_copilot.py`/`copilot_audit.py`/`copilot_evaluation.py`) sits over the already-governed
decision documented above and **cannot change it**: no score, threshold, `RECOMMENDED`/
`CONTESTED`/`MONITOR` status, resource allocation, or model/policy promotion is ever written by
the Copilot. It only reads a deterministic, allowlisted, cited evidence packet built from the
same persisted `scored_orders.csv`/`metrics.json`/`run_manifest.json` this card already describes,
and answers a fixed order-question catalog or fixed portfolio-question catalog -- never
unrestricted DataFrame/SQL/code execution.

- **Two interchangeable providers, one schema.** A deterministic fallback
  (`copilot_fallback.py`) produces the same structured, cited JSON response with plain Python --
  no network, no key. An optional live path (`llm_copilot.py`) calls the official `openai` SDK's
  Responses API (Structured Outputs, strict JSON schema) when `OPENAI_API_KEY` is configured
  (`OPENAI_MODEL` default `gpt-5-mini`; `OTIF_LLM_MODE=auto|live|fallback`). Any import, timeout,
  API, parsing, or validation failure falls back to the deterministic response -- never a blank
  page or ungrounded free text.
- **Validated, not just prompted** (`copilot_validation.py`): rejects unknown citation IDs,
  requires a citation per factual claim, requires `preserves_persisted_decision: true`, rejects a
  response whose text asserts a different decision status than the persisted one, rejects
  non-finite/oversized output, and strips unsupported HTML/URLs.
- **Evaluated mechanically, not with unearned quality claims** (`copilot_evaluation.py`, `uv run
  otif-copilot-eval`): a deterministic representative-order set (high-risk inventory miss,
  timing-driven miss, multi-cause, contested action, low-confidence, safe/monitor, unknown-cause)
  is checked for citation validity, decision preservation, section completeness, fallback success,
  and latency/token usage, written to `artifacts/copilot_evaluation.json`. **No BLEU, factuality,
  or human-preference score is claimed** -- there is no human-labeled explanation dataset for this
  prototype.
- **Audited** (`copilot_audit.py`): every request appends one JSON line to `copilot_audit.jsonl`
  with request ID/timestamp, scope/query type, provider/model/mode, an evidence-packet hash,
  latency, token usage when reported, validation status/fallback reason, and cited fact IDs --
  never the API key, the full response text, or hidden reasoning.

## Honest limitations

- **Bayesian standalone quality remains below XGBoost** (median PR-AUC 0.475 vs. 0.700):
  the mechanism network prioritizes interpretable propagation, but contributes 10–30% when
  validation shows practically equivalent calibration under the recall guardrail.
- **The LATE_DELIVERY mechanism is a weak predictor in this synthetic twin**: PR-AUC 0.126
  median against a ~3.8% held-out base rate (roughly 3x lift, but far weaker than
  IN_FULL_FAILURE's 0.417 against its ~14% base rate). Most modeled OTIF misses in this
  digital twin are dominated by quantity shortfall rather than timing delay; this is a
  genuine, measured property of the fitted network on this data, not a claim adjusted to
  look better, and it means LATE_DELIVERY-side pathway/intervention detail should be read
  with lower confidence than IN_FULL_FAILURE-side detail.
- **Causal-node consistency is moderate, not validation**: median agreement with the
  retrospective rule-derived primary cause is about 50% on held-out misses with active
  evidence. Remaining disagreement reflects a real distinction between posterior-risk
  contribution, upstream-priority attribution, and a largest-delay heuristic. Structural
  intervention outputs must not be presented as validated treatment rankings.
- **Structural interventions and evidence attribution are not causal-effect estimates**:
  `do(node=value)` scenarios are exact computations under this fixed, fitted network's
  assumptions ("fixed-structure scenario analysis"), and evidence attribution is a
  leave-one-out sensitivity measure, not SHAP and not an identified causal effect.
- **Synthetic data only**: entity counts (12 vendors, 5 DCs, 15 lanes, 40 customers, 120
  SKUs) are modest; real deployments would need materially more data-quality validation,
  monitoring, and governance before any production use.
- **No causal-effect claim**: SHAP factors and Bayesian pathways are associations;
  intervention "avoided misses"/savings in the operations loop are simulated estimates,
  never observed causal impact.
- **Drift thresholds are illustrative**: PSI/score/missingness/rate thresholds were
  tuned for a demonstrable-but-not-thrashing daily replay on this specific twin, not
  validated against a real production monitoring baseline.
- **`ORDER_CAPTURE_CORRECTION` is a structural no-op in this twin**: `data.py`'s
  ship/transit timing depends only on `order_date`, never `capture_delay_hours`, so this
  action can never mechanically change a delivered timestamp/quantity, verified across
  every benchmarked seed. It is retained for cause-coverage completeness, and the
  Decision Value Lab's headline metrics are honest about it contributing zero benefit.
- **`CURRENT_POLICY` beats every deployable baseline at the primary 50%-capacity
  headline, and the acceptance gate is measured and reported as passed**: at this
  twin's *default* (100%) resource capacities, capacity is rarely contested for
  single-action baselines, so on 3 of the 5 benchmarked seeds every single-action policy
  -- including plain random selection -- lands on numerically identical results. The
  value-aware `CURRENT_POLICY` changes *what* gets decided (candidate actions, not just
  ranking order), so it diverges from the single-action baselines' identical results even
  on those seeds, and wins 4 of 5 seeds against `RANDOM_AT_CAPACITY`,
  `HIGHEST_RISK_AT_CAPACITY`, and `SINGLE_CAUSE_PRIORITY_BASELINE` at the Stage 1 primary
  50%-capacity scenario, beating the win threshold (`>=3/5`). Its median (9.964) beats
  the shared baseline median (9.174) by +8.6%, it wins every baseline at 25% and 100%
  capacity too, and it improves regret-vs-oracle and avoidable-miss coverage without an
  action-precision collapse. The one honest caveat: a Bayesian-ablation diagnostic shows
  the policy's Bayesian structural-reduction term does not currently add value over its
  own simpler fallback in this twin (0 of 5 seeds). See "Decision Value Lab" above for
  the full per-scenario table and per-seed win/tie/loss counts, and "Governance (Stage 2)"
  below for how the promotion gate uses this exact finding.
- **The Bayesian top intervention does not win every counterfactual-ranking axis**: it
  has a *lower* raw top-action-agreement rate than the strongest-signal heuristic in
  every benchmarked seed, and a *higher* (worse) mean value regret than the
  strongest-signal heuristic in 2 of 5 seeds, even though it beats a random feasible
  action's regret in all 5 and beats the strongest-signal heuristic's regret in 3 of 5.
- **Decision Value Lab results remain simulator-evaluation, not real-world causal
  effect**: every avoided-penalty figure is computed by replaying the twin's own fixed
  response-probability/magnitude formulas under common random numbers -- a measured,
  falsifiable number *within this synthetic simulation*, not evidence of a real
  operational treatment effect. The oracle policy is an explicit, unattainable ceiling,
  never a deployable recommendation.
- **Every real retrain challenger was held in the canonical 90-day replay, and the
  rolling PR-AUC SLO fails as a direct, reported result**: the promotion gate correctly
  kept the initial, better-measured champion active rather than promoting any of the 8
  real retrains (each regressed held-out PR-AUC beyond tolerance on this short window's
  necessarily small/noisy test splits) -- this is the gate working as intended, not a
  defect, but it means the active model's realized quality was never refreshed with more
  recent data during this specific replay, and the honestly-reported cost is a failed
  rolling-PR-AUC SLO (0.386 vs. 0.55 target). A longer replay window or a larger matured
  population per retrain would likely narrow this test-split noise.
- **Real per-retrain promotion checks reuse one reference policy-value number** (Stage
  1's measured 5-seed median at 50% capacity for the deployed `CURRENT_POLICY`) rather
  than re-running the full rolling-origin policy-value lab for every retrain, which would
  be prohibitively expensive across a 90-day replay; this makes the policy-value gate
  non-discriminative between real retrains (they share the same policy formula) by
  construction. Only the demo governance-lifecycle scenario (see "Governance (Stage 2)"
  above) compares genuinely different policy-value numbers (single-cause baseline vs.
  `CURRENT_POLICY`, and with vs. without the Bayesian term).
- **The observational decision-ledger cohort report is not a causal estimate**: accepted
  vs. monitored miss-rate differences reflect which orders this replay's simple
  cause-lookup policy chose to prioritize, not a measured treatment effect -- the report
  is explicitly labeled `observational_not_causal` and is a separate, weaker claim than
  the Decision Value Lab's exact, common-random-number potential-outcome policy value.
- **The AI Copilot's validation guards against unsupported claims; it does not prove an
  upstream LLM can never hallucinate.** The citation/decision/finite/size checks in
  `copilot_validation.py` make ungrounded output visible and force a fallback rather than
  presenting it as grounded, but they cannot verify that a live model's phrasing is a
  perfectly faithful paraphrase of a cited fact, only that every cited fact ID exists and
  the persisted decision is preserved. The evaluation harness reports only mechanical
  metrics (citation validity, decision preservation, completeness, fallback success,
  latency/tokens) -- never a claimed factuality, BLEU, or human-preference score, since no
  labeled explanation dataset exists for this prototype.
