    Section 2 · Data foundation

    The noisy digital twin, and production-shaped source adapters

    There is no real ERP/WMS/TMS in this prototype — there is a digital twin (a term for "a simulator built to behave like the real thing"): it plays out every order's life stage by stage — capture, vendor readiness, warehouse, transport, customer delivery — with stable vendor/lane/DC traits, seasonality, correlated disruption shocks, and realistic noise. Orders are never pre-labeled as "will miss"; the OTIF (on-time-in-full) outcome falls out of accumulated delay and shipped quantity, the same way a real fulfillment network would produce it.

    What goes in

    Vendor/SKU/DC/lane/customer master traits, a demand/seasonality curve, randomly-timed disruption shocks, and a scripted late-horizon drift epoch of elevated vendor disruption.

    What happens

    Each order's lifecycle is simulated stage by stage. ~3% of events never post at all; exception codes log with ~85–90% probability even when a real delay crossed its threshold; a capture-time inventory shortfall has a documented chance of being backfilled before shipment.

    What comes out

    Canonical order/line/event tables (model-facing) plus a separate ground-truth set (simulator_truth.csv, line_truth.csv, shocks.csv) used only for scoring the model afterward — never as a feature.

    Why it matters

    A generator that pre-decides who misses and forces matching evidence would make the whole prediction problem fake. This twin can produce a safe order with scary-looking signals, and a missed order with almost no early evidence — like real supply chains.

    Measured Across a 5-seed benchmark (2,500 orders/seed) the OTIF miss rate lands at a median 17.5% (range 15.2%–20.2%) — inside the intended 15–25% band on every seed tested.

    Five reserved scenarios, every seed

    Regardless of random seed, the last five orders of every run are deterministic named scenarios, so the story below is always reproducible: multi-cause propagation, two orders contesting shared recovery capacity (our running example), a single-line stockout inside a multi-line order, and a genuinely unexplained miss with zero corroborating evidence.

    Then the same data, through production-shaped pipes

    Separately, adapters.py and service_contracts.py wrap typed local-CSV ERP/WMS/TMS/SRM adapters that reconstruct the same canonical tables — but only after redacting any event whose timestamp hasn't happened yet as of the requested snapshot. A typed ScoreRequest/ScoreResponse contract and an idempotent decision sink stand in for what a real integration boundary would look like.

    What goes in

    The same order/as-of snapshot, requested once directly against the in-memory dataset and once through the adapter/service boundary (CSV round-tripped tables).

    What happens

    Both paths run the identical point-in-time feature contract. Offline/batch parity is checked by comparing feature vectors and scores byte-for-byte.

    What comes out

    parity_check.json: 15 of 15 orders checked, 0 mismatches, passed: true.

    Why it matters

    A model that scores differently depending on which door you walked through is not trustworthy in production. This proves — for this prototype's scope — that it doesn't.

    Honest scope: these are typed Python interfaces plus local-CSV fixtures — production-shaped, but deliberately not a web framework, database, message broker, or cloud deployment. No live ERP/WMS/TMS/SRM is connected anywhere in this prototype.

    Section 3

    Data contracts and quality checks

    Before anything is trained or scored, every table is validated: schema (right columns, right types), referential integrity (every order line points to a real order, every order to a real vendor), and logical checks (quantities can't be negative, dates can't be backwards). This fails fast rather than silently producing garbage features.

    What goes in

    Raw canonical tables — orders, order lines, vendors, DCs, lanes, customers, SKUs, capacity snapshots, events.

    What happens

    validation.py checks schema, referential links, and logical constraints; the monitoring layer separately tracks completeness, uniqueness, and referential health as ongoing metrics, not just a one-time gate.

    What comes out

    A pass/fail contract-failure count, plus per-table completeness and uniqueness scores persisted in the monitoring report.

    Why it matters

    Garbage in, confidently-wrong risk scores out. Catching a broken join or a missing table here is cheaper than debugging a bad prediction later.

    Measured in the latest 90-day governed replay: 0 contract failures, all referential checks pass, table completeness ranges 81.2% (events — expected, since ~3–15% of events never post in the twin) to 100% (orders, lines, vendors, DCs, lanes, customers, SKUs, capacity snapshots).

    Section 4

    The as-of timestamp: why the model can never see the future

    Every feature is computed as of an explicit as-of timestamp. That is the single contract used both for historical training snapshots and for the live daily scoring loop. Leakage is the failure mode this guards against: a feature that secretly encodes the final outcome.

    What goes in

    Operational fields/events already filtered to event_timestamp <= as_of_timestamp — vendor delay/exception, warehouse/transport exceptions, order-capture fields, the DC capacity snapshot as of that date.

    What happens

    Rolling vendor/DC/lane/customer/SKU history is computed in three matured windows (30-day, 90-day, all-time). A timestamp-group chronological split keeps orders with identical timestamps on the same side of a train/validation/test boundary.

    What comes out

    ~100 point-in-time features per order, none of which read the simulator's latent disruption cause directly.

    Why it matters

    Without this, a benchmark number is meaningless — it would be measuring the model's ability to peek at the answer, not to predict.

    Tested tests/test_features.py proves: leading-signal features are not present on the raw generator output, are not a lossless proxy for the ground-truth cause, and mutating every future event/outcome cannot change an earlier as-of snapshot's row.

    Section 5 · Prediction engine

    XGBoost risk prediction and calibration

    XGBoost is a gradient-boosted decision-tree model — it builds many small decision rules and combines them into one score. It is trained on ~100 point-in-time features to output one number: the probability this order misses OTIF. Calibration (here, Platt scaling, fit on the validation split) makes that number mean what it says, not just rank orders correctly.

    What goes in

    ~100 point-in-time features per order (vendor/DC/lane/customer/SKU rolling history, freshness, missingness, remaining slack, DC utilization trend, line-evidence aggregates).

    What happens

    Gradient-boosted binary classification, then Platt-scaling the raw scores on the validation split so probabilities are meaningful, not just ranked.

    What comes out

    A calibrated OTIF-miss probability per order (xgb_risk_score).

    Why it matters

    This is the model doing the heavy lifting — it is the single largest predictive contributor to the fused score (see Fusion below).

    Measured (5-seed median): XGBoost-only PR-AUC 0.700 (range 0.533–0.771). PR-AUC is the headline separability metric used throughout this system because OTIF misses are the minority class.

    SHAP: which features pushed this score up or down

    SHAP explains one order's score by attributing it to individual features. This prototype uses SHAP when available, with a deterministic local-perturbation fallback otherwise (same idea: nudge one feature, see how much the score moves).

    Explicit limitation: every SHAP/perturbation factor is labeled association_not_causation. "Affected line count contributed +0.18 to this score" means the model found that feature predictive — it is not a claim that reducing it would cause the score to drop by 0.18.

    Real example For order O002497 (our running example below), the top SHAP-style factors are: affected_line_count (+0.179), sku_rolling_shortfall_rate_mean_90d (+0.076), max_line_risk_evidence (+0.068), ship_delay_hours (+0.067) — all pushing risk higher.

    Section 6

    The 10-node Bayesian mechanism graph: timing vs. quantity

    A Bayesian network here is a diagram of operational stages wired together the way OTIF actually breaks: "on time" AND "in full" are two separate failure mechanisms. This network has exactly 10 binary nodes and splits the single old "OTIF miss" idea into LATE_DELIVERY (a timing problem) and IN_FULL_FAILURE (a quantity problem), both feeding OTIF_MISS.

    What goes in

    Training-only resolved history: each operational stage's incident flag (including disruptions absorbed without an OTIF miss), plus each order's own on-time/in-full outcome for the two mechanism nodes.

    What happens

    Smoothed-count probability tables are fit per node. A node only becomes "hard evidence" once its stage has genuinely been observed as of the as-of timestamp; unobserved stages (including the two mechanism nodes themselves) are mathematically averaged out (marginalized) via exact inference — pgmpy when available, or a brute-force enumeration verified numerically identical to it.

    What comes out

    Two separate posterior probabilities per order: P(IN_FULL_FAILURE) and P(LATE_DELIVERY), the active evidence nodes, and the mechanism route(s) they feed.

    Why it matters

    A single flat "risk score" hides whether the danger is a quantity shortfall or a timing delay — these need different interventions. Splitting the mechanism tells the planner which kind of problem they're actually facing.

    Real example On order O002497: P(IN_FULL_FAILURE) = 99.5% vs. P(LATE_DELIVERY) = 9.6% — overwhelmingly a quantity problem, not a timing one. Measured 5-seed medians: IN_FULL_FAILURE PR-AUC 0.417 (vs. its own ~13–15% base rate) and LATE_DELIVERY PR-AUC 0.126 (vs. its own ~3.8% base rate) — both real, both modest, both reported honestly rather than hidden.

    Section 7

    Structural intervention scenarios — and their limitation

    For every scored order, the Bayesian network can also answer a "what if we fixed this?" question: do(node = 0) replaces one node's fitted behavior with a fixed value and severs its parents' influence on it, then recomputes the OTIF-miss posterior. This is an exact computation under the network's fixed structure — it is not a proven treatment effect from a real experiment.

    Fixed-structure scenario analysis — not a proven treatment effect. These numbers never feed the XGBoost score, the fused score, or the operational decision. They are diagnostics, surfaced only in the Causal Intelligence Studio view.

    Try the interactive version of this in "What if?" below — it uses this exact order's real, persisted intervention scenarios.

    What goes in

    The order's active evidence nodes (only cause nodes observed strictly at/before decision time) and the fitted network.

    What happens

    One do(node=0) scenario per active evidence node, plus one combined-mitigation scenario, always computed via brute-force enumeration (never through the observational-query shortcut).

    What comes out

    Baseline/post-intervention posteriors, absolute/relative risk reduction, the mechanism routes affected, and the assumed operational action — each explicitly labeled.

    Why it matters

    It can reveal that "fixing" the obvious root cause does nothing once a downstream effect is already locked in — a genuinely useful planning signal, even without being a proven causal estimate.

    Section 8

    Risk fusion, and why XGBoost stays primary

    The final risk score blends XGBoost and Bayesian: combined_risk_score = weight × xgb_risk_score + (1 − weight) × bbn_risk_score. The weight is chosen on the validation split only, from a fixed 10%-increment grid (0%, 10%, 20% … 100%) — never a learned stacking model. A candidate must clear a recall floor and land within 0.002 Brier score of the best candidate; among ties, the rule prefers more Bayesian contribution.

    What goes in

    XGBoost scores and Bayesian posteriors on validation, at every weight from 0.0 to 1.0 in 0.1 steps.

    What happens

    Score each candidate weight's Brier score (calibration quality) and recall at the top-capacity operating point; keep only candidates within the guardrails; prefer the most Bayesian-weighted survivor.

    What comes out

    One chosen blend weight, persisted with its full comparison table — regardless of which candidate won.

    Why it matters

    XGBoost is simply the stronger predictor here (PR-AUC 0.700 vs. Bayesian's 0.475), so fusion typically lands mostly on XGBoost — but the Bayesian network still contributes some validated lift, and always earns its seat by winning a real comparison, not by assumption.

    Measured on the 5-seed benchmark, validation selected 20–30% Bayesian contribution; the canonical seed-42 run selected 10% Bayesian / 90% XGBoost. Fused median PR-AUC 0.705 (range 0.535–0.763) — a small but real lift over XGBoost alone.

    Section 9

    Affected SKU / line evidence: which lines, not just which order

    A risk score tells a planner "this order is in trouble." Line evidence tells them which lines inside it are the actual problem — critical for partial-shipment decisions. It is computed only from fields known at capture time: initial available-to-promise (ATP) allocation, inventory snapshot, SKU criticality, and the order's own point-in-time vendor-exception signal.

    What goes in

    Per-line ATP allocation ratio, inventory snapshot, SKU criticality tier, and the order-level vendor-exception signal — all capture-time-safe.

    What happens

    Each line gets an evidence-strength score and a "likely affected" flag, aggregated into order-level features (worst line shortage ratio, affected line count, critical-SKU share, line quantity concentration).

    What comes out

    A ranked list of likely-affected SKUs per order, each with its allocation-gap ratio and evidence strength.

    Why it matters

    Knowing "SKU080 is 14.6% short" is actionable in a way "this order is risky" alone is not — it tells the planner exactly where to intervene.

    Section 10 · Decision & action

    Value-aware action selection: the exact formula

    Older single-cause policies gave every eligible order the one action its primary cause implied. The current, deployed policy instead considers every point-in-time-feasible action for each order and ranks all order-action pairs with one transparent formula — no learned weights, no access to any "did it actually work" field.

    expected_benefit = estimated_penalty_exposure × structural_reduction × execution_feasibility
value_density     = expected_benefit / normalized_resource_fraction

    What goes in

    estimated_penalty_exposure (order value × penalty rate × combined risk); structural_reduction (the Bayesian do-intervention's relative risk reduction if one exists for this cause, else a documented 60% fallback scaled by match confidence); execution_feasibility (remaining slack, an action-specific resource trait, and Bayesian evidence coverage); and normalized_resource_fraction (the action's resource demand ÷ that pool's default daily capacity).

    What happens

    Every active point-in-time signal that maps to a feasible action becomes a candidate; each candidate gets a value density; the highest-density candidate per order becomes its assigned action, and the same score ranks orders against each other for scarce capacity.

    What comes out

    One chosen action per eligible order (e.g. VENDOR_ESCALATION, INVENTORY_REALLOCATION, WAREHOUSE_EXPEDITE, ALTERNATE_TRANSPORT, APPOINTMENT_COORDINATION), ranked by expected avoided penalty per unit of scarce capacity consumed.

    Why it matters

    Under this twin's realistic capacity limits, this changes what gets decided, not just whose turn is first — and it is measured, not assumed, to beat simpler baselines (see the Policy comparison below).

    Every weight and fallback listed above was fixed in code before any benchmark ran — none were retuned after seeing results.

    Section 11

    Capacity allocation: RECOMMENDED, CONTESTED, MONITOR

    Real recovery resources are scarce: DC recovery units, lane alternate capacity, vendor escalation slots, customer appointment slots. Every value-ranked order-action pair competes for its resource pool, allocated greedily by priority within each day's capacity.

    What goes in

    The eligible order-action pool (fused risk ≥ threshold and a feasible mapped action), ranked by value density, and each resource pool's daily capacity.

    What happens

    Orders are granted capacity in priority order until a pool runs out; anyone who loses out is marked CONTESTED and the competing order(s) are named; everyone below the fused threshold is MONITOR.

    What comes out

    One of three statuses per order: RECOMMENDED (capacity granted), CONTESTED (eligible but resource-blocked, competitor named), or MONITOR (below the decision threshold).

    Why it matters

    A recommendation nobody can actually execute is worse than no recommendation — naming the contention makes the resource conflict visible to a human planner instead of hiding it.

    Measured canonical seed-42 run — 495 held-out test orders: 62 RECOMMENDED, 41 CONTESTED, 392 MONITOR.

    Section 14

    The 10% exploration carve-out

    If the deployed policy only ever acted on its own top-ranked guesses, it could never learn whether its ranking was actually right — every order it chose not to act on would remain a permanent unknown. So roughly 10% of allocated capacity is deliberately set aside for near-equal eligible orders regardless of rank, using seeded randomization.

    What goes in

    The pool of eligible order-action candidates within a resource pool, near the acceptance cutoff.

    What happens

    Discrete slot pools (vendor/customer) split whole-slot counts via seeded stochastic rounding, so every candidate's exact marginal selection probability is logged. Continuous pools (DC/lane) fill the explore stage up to actual remaining capacity and log a pool reservation ratio instead of a single-order probability where no closed form exists.

    What comes out

    A small, transparent, logged fraction of "explore" assignments alongside the main "exploit" (highest value-density) assignments.

    Why it matters

    Without exploration, you can never build a fair dataset to check whether your ranking formula was correct — you'd only ever see outcomes for orders you already believed were highest-value.

    Section 15 · Operating loop

    The daily operations replay

    A local simulation of an actual operating day: train an initial model on historical data, then for each simulated day, score every still-open order as of that day, allocate that day's resource capacities, persist the queue, close resolved orders, derive their actual cause, log feedback, check for drift, and retrain on a schedule or when drift triggers it.

    What goes in

    A 2,500-order twin, replayed across 90 simulated days, one shared as-of snapshot per day.

    What happens

    Daily scoring → capacity allocation → order closures → cause derivation → feedback → drift check (PSI, score-distribution shift, missingness change, recent OTIF-rate change) → retrain on cadence or trigger.

    What comes out

    A versioned model registry, daily decision queues, and every Stage-2 governance artifact (ledger, monitoring, promotion events) described below.

    Why it matters

    A one-shot benchmark can't show whether a system stays healthy over time, retrains sensibly, or degrades quietly — this replay is the closest local approximation to "running it for real."

    Measured the canonical 90-day replay trained 9 model versions (1 initial + 8 cadence/drift-triggered retrains). Measured local-only runtime: mean scoring 1.38s/day, mean retrain 3.21s across 8 retrains — explicitly not a production latency claim.

    Section 16

    Run manifests, checksums, and offline/batch parity

    Every run — pipeline, benchmark, or operations replay — writes a run manifest: the git commit, dependency versions, config/policy/schema versions, input-table fingerprints, a feature-schema hash, the train/validation/test date windows, and a SHA-256 checksum of every other artifact the run wrote.

    What goes in

    Everything the run itself produced, plus the environment it ran in (git SHA, dirty-tree flag, package versions).

    What happens

    A deterministic content ID is computed that ignores timestamps/run-instance/output-path — so two runs of identical seed/config/code/model-facing artifacts, even on different machines or at different times, get the same content ID. verify_manifest recomputes every checksum and flags any mismatch.

    What comes out

    A run_manifest.json with per-file checksums and a verified/tampered status.

    Why it matters

    If a benchmark number ever looks suspicious, the manifest tells you exactly what produced it, and whether any artifact was altered afterward.

    Measured, latest verified run the canonical pipeline run (seed 42, 2,500 orders) checksummed 29 artifacts, fully verified. The latest 90-day governed operations replay checksummed 121 artifacts, also fully verified. The same replay's offline/batch parity check: 15 of 15 orders matched byte-for-byte between the in-process feature builder and the adapter/service boundary — mismatched_order_ids: [].

    Section 17

    The Decision Value Lab: measuring, not assuming, intervention value

    The deployed decision table uses a fast, fixed 60%-effectiveness assumption for "how much does this action help" — a transparent heuristic, not a measurement. The Decision Value Lab is a separate laboratory that replays every feasible action through the twin's own lifecycle mechanics with a shared random draw per (seed, order, action) — so every policy under test sees the exact same potential outcome for the same order/action, an apples-to-apples comparison.

    Every order is scored via rolling-origin, chronological cross-fitting: the calendar splits into 5 folds, the first is warm-up history (excluded), and every later fold is scored only by a model trained strictly on history before it — genuinely out-of-sample, no fold ever influences its own scoring.

    Eight policies are compared at the identical out-of-sample orders, the identical potential outcomes, and identical daily capacities: NO_ACTION, RANDOM_AT_CAPACITY, HIGHEST_RISK_AT_CAPACITY, HIGHEST_FINANCIAL_AT_CAPACITY, STRONGEST_SIGNAL_HEURISTIC, SINGLE_CAUSE_PRIORITY_BASELINE (the older transparent risk×tier×value ranking), CURRENT_POLICY (the deployed value-aware policy), and ORACLE_EVALUATION_ONLY — an unattainable, evaluation-only ceiling that may choose any action, used only to compute regret, never a deployable recommendation.

    25% / 50% / 100% capacity-stress comparison

    At the twin's full (100%) default capacity, resource pools rarely bind — most policies simply accept every eligible order, leaving little for ranking to improve. So every policy is re-run at three capacity-stress multipliers applied uniformly to every resource pool. 50% capacity is the Stage-1 primary headline — the acceptance gates below are measured there.

    Section 18 · Governance

    Decision ledger and observational outcome cohorts

    Every decision the operations replay makes — feasible actions considered, the one chosen, the one rejected, resource state before/after, execution status — is logged to an append/upsert, idempotent ledger, one row per open order scored per day. Once an order's real OTIF outcome matures, it is reconciled against that logged decision.

    What goes in

    Every daily decision from the 90-day replay, plus each order's eventual matured outcome.

    What happens

    Decisions are grouped into cohorts — ACCEPTED, MONITORED, REJECTED — and each cohort's realized miss rate and mean penalty are compared.

    What comes out

    An observational cohort report, explicitly flagged observational_not_causal: true.

    Why it matters

    It would be easy to misread "accepted orders miss more often" as "acting made things worse." It doesn't mean that — see the limitation below.

    Measured latest governed replay: 6,226 decisions logged, 5,923 matured (303 still open at replay end). ACCEPTED orders (716 matured): 57.96% miss rate, mean realized penalty $113.54. MONITORED orders (5,206 matured): 10.08% miss rate, mean realized penalty $13.33.

    Not a causal effect. This replay closes every order against its own pre-generated outcome regardless of the decision made — action does not yet feed back into the replayed lifecycle here. The gap above reflects which orders the policy chose to prioritize (correctly, the higher-risk ones), never a measured effect of acting on them. The Decision Value Lab above remains the only causally-interpretable (common-random-number, potential-outcome) policy value in this codebase, kept deliberately separate from this ledger.

    Section 19

    Champion/challenger gate: PROMOTED, HELD, ROLLED_BACK

    Every new model version is a champion/challenger candidate. evaluate_promotion compares it against the current champion on PR-AUC, Brier, calibration error, recall, alert rate, drift-regime quality, and policy value at 50% capacity — each against an explicit, fixed tolerance. Any failed check holds the challenger; the active-model pointer is only updated atomically on a real promotion or an explicit rollback, and every event is appended to a log that is never rewritten.

    Two lifecycles — kept explicitly distinct

    The real 90-day retrain lifecycle and the demo governance-lifecycle scenario are different things, and the artifacts label them differently so they can never be confused.

    Real retrain lifecycle (measured, all 8 challengers held)

    Section 23 · Honesty

    What is real vs. simulated

    Fully working, locally

    The digital twin, feature pipeline, XGBoost model, SHAP explanations, the 10-node Bayesian network and exact inference, fusion, line evidence, resource allocation, the value-aware policy formula, run manifests/checksums, offline/batch parity, the decision ledger, champion/challenger promotion logic, rolling monitoring/SLOs, and the Streamlit UI all run end to end, locally, on real (synthetic) data — every number on this page came from actually running this code.

    Simulated / evaluation-only

    All underlying data is a synthetic digital twin, not real ERP/WMS/TMS data. The Decision Value Lab's policy values are measured against this twin's own mechanics, not a real supply chain. The oracle policy is an unattainable evaluation-only ceiling. The demo governance lifecycle is built from Stage 1's own measured numbers, distinct from the real per-day retrain lifecycle.

    Causal limitations (read this one)

    Bayesian do-interventions are fixed-structure scenario analysis, not proven treatment effects. SHAP/perturbation factors are associations, not causes. Evidence attribution is leave-one-out sensitivity, explicitly not SHAP and not a causal estimate. The observational cohort report is non-causal by construction (the replay closes every order against its pre-generated outcome regardless of the decision made). None of these limitations are hidden — every one is labeled in the persisted artifacts themselves.

    Production interfaces, mocked

    ERP/WMS/TMS/SRM adapters are typed local-CSV implementations of a real interface contract — proven to have offline/batch parity, but deliberately not a web framework, database, message broker, or cloud deployment. The LLM-style narrative layer is a deterministic template, not a live LLM.

    Section 24

    How to explain it in 90 seconds

    "This system takes a noisy, realistic supply-chain simulation and, for every open order, produces one risk score, one plain-language explanation, and one resource-aware recommendation. The score blends a gradient-boosted model with a small Bayesian network that splits 'late' from 'short' failures, so we know not just that an order is at risk but what kind of risk it is. A value-aware formula then ranks every feasible action by expected avoided penalty per unit of scarce recovery capacity — so the busiest vendor slot goes to the order that needs it most, and the loser is told explicitly who it's competing with.

    Every decision is logged to an auditable ledger and reconciled against real outcomes. A promotion gate compares any new model version against the current champion on measured quality and decision value, and only promotes if it's genuinely better — in our 90-day replay, it correctly held all eight retrain attempts and one honestly-labeled Bayesian-enhanced challenger that looked good in isolation but regressed against the actual champion. Every number here — PR-AUC, policy value, ledger counts, SLA pass/fail — comes from a real regenerated run, and every causal-sounding claim is labeled as association or fixed-structure scenario analysis, never asserted as proven cause and effect."

    Section 25

    Glossary

    As-of timestamp
    The single point in time a snapshot is "taken" — only data that existed at or before this timestamp may be used as a feature.
    Bayesian network
    A model that represents variables as a network of probabilistic dependencies and computes exact probabilities by combining evidence across the whole network.
    Calibration
    Adjusting raw model scores so "70% risk" really means about 70 out of 100 similar orders miss — not just a correct ranking.
    Champion/challenger
    A promotion-gate pattern: the active model ("champion") stays in charge until a new candidate ("challenger") proves it is measurably better on every required check.
    Content ID
    A deterministic hash of a run's code/config/data that ignores timestamps and output paths — identical runs always get the identical ID.
    Digital twin
    A simulator built to behave like a real system closely enough to test decisions against it.
    do(node = value)
    A causal-inference operation that simulates an external action overriding one variable, severing its normal causes, then asks what happens downstream — different from just observing that value.
    Drift
    A statistical change in the underlying data over time (e.g., vendors becoming less reliable) that can make an old model's predictions stale.
    Evidence attribution (leave-one-out)
    How much a Bayesian posterior would change if one observed cause were withheld — a sensitivity measure, explicitly not SHAP and not a causal effect.
    Leakage
    A model accidentally learning from information that would not actually have been available at prediction time.
    Manifest
    A machine-checkable receipt of exactly what produced a run — code version, dependencies, config, and per-file checksums.
    Marginalize
    Computing a probability by properly averaging over everything not yet known, rather than guessing or ignoring it.
    Observational (not causal)
    A comparison of outcomes across groups that were not randomly assigned — differences may reflect who was selected, not the effect of any action.
    Offline/batch parity
    Proof that two different code paths that should agree (e.g., direct scoring vs. scoring through a service boundary) actually produce identical output.
    OTIF
    On-Time-In-Full — an order only counts as OTIF if it is both on time and shipped complete; missing either counts as an OTIF miss.
    PR-AUC
    Precision-Recall Area Under Curve — how well a model separates true positives from true negatives, especially useful when the positive class (here, OTIF misses) is rare.
    SHAP
    SHapley Additive exPlanations — attributes one prediction to its input features, showing how much each pushed the score up or down for this specific case.
    SLO
    Service-Level Objective — a transparent, pre-committed operational target that is checked automatically and either passes or fails.
    Value density
    Expected avoided penalty from an action, divided by how much scarce resource capacity that action consumes — lets very different actions be ranked on one scale.
